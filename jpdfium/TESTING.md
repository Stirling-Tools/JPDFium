# JPDFium Redaction Testing Guide

## Overview

This guide explains the automated testing infrastructure for verifying PDF redaction
correctness. The test suite ensures that the Object Fission redaction algorithm removes
targeted text while preserving non-targeted content and layout.

## Quick Start

```bash
# Run all integration tests (includes corpus redaction test)
./gradlew :jpdfium:integrationTest

# Run only the corpus redaction test
./gradlew :jpdfium:integrationTest --tests "stirling.software.jpdfium.CorpusRedactTest"

# Run the coordinate-level redaction tests
./gradlew :jpdfium:integrationTest --tests "stirling.software.jpdfium.redact.ObjectFissionCoordinateTest"
```

After running, open the HTML report:
```
samples-output/corpus-redact-report/index.html
```

## Test Architecture

### Test Classes

| Class | Purpose | Scope |
|-------|---------|-------|
| `CorpusRedactTest` | Corpus-wide visual + structural testing | 66 PDFs (30+ downloaded + 20 local) |
| `ObjectFissionCoordinateTest` | Character-position-level precision | 84 generated test PDFs |
| `VisualRedactTest` | Pixel-level redaction accuracy | minimal.pdf |
| `ObjectFissionRedactTest` | API smoke tests + option coverage | minimal.pdf |
| `RealPdfIntegrationTest` | Basic document operations | minimal.pdf |


## Structural Failure Detection

The `CorpusRedactTest` applies three independent checks per page to detect redaction
problems. Each check has a threshold that, when exceeded, triggers a specific severity:

### 1. Character Loss Ratio (FAIL if > 40%)

Compares the number of extractable characters before and after redaction. The redact
words are common English words ("the", "and", "for", "that", "this", "with"), so some
character loss is expected. However, losing more than 40% of all text indicates the
algorithm damaged content beyond the target words.

**Example failure:** `issue918.pdf` -- 100% char loss (1609 to 0 chars). All text was
removed, not just the target words. This is a genuine Object Fission bug on that PDF.

### 2. Non-Target Word Survival (FAIL if any missing)

Samples up to 5 words from the original text that are NOT in the redact target list
(words with 4+ alphabetic characters, deduplicated). After redaction, each sampled word
is searched in the remaining text. If any disappear, the redaction corrupted non-target
content.

**Example:** In `issue918.pdf`, the words "Ideology", "petty", "hides", "survival",
"speech" all disappeared despite not being in the redact list.

### 3. Visual Pixel Change Ratio (WARN if > 15%)

Renders pages at 72 DPI before and after redaction, counts changed pixels. A high
pixel change ratio does not automatically mean corruption -- it can be legitimate when
redacting common words from text-heavy documents. This check issues a WARNING to draw
attention for manual inspection.

**Example:** `tracemonkey.pdf` shows 17-21% pixel changes across 3 pages. This is
expected because "the", "and", "for" etc. appear frequently in the academic paper.
The char-loss ratio stays within bounds, so it passes.

## Severity Levels

| Severity | Meaning | JUnit Result |
|----------|---------|--------------|
| **FAIL** | Structural corruption detected (char loss > 40% OR non-target words missing) | Test fails |
| **WARN** | High visual impact (>15% pixels changed) but no structural corruption | Test passes |
| **PASS** | All checks within thresholds | Test passes |
| **SKIP** | PDF could not be opened or analyzed (encrypted, malformed, etc.) | Test passes |

## Thresholds

All thresholds are defined as constants in `CorpusRedactTest.java` and can be adjusted:

```java
MAX_CHAR_LOSS_FRACTION       = 0.40   // >40% char loss = FAIL
WARN_PIXEL_CHANGE_FRACTION   = 0.15   // >15% pixels changed = WARN
NON_TARGET_SAMPLE_SIZE       = 5      // number of non-target words to verify
MAX_PAGES_PER_PDF            = 3      // pages tested per PDF (for performance)
MAX_LOCAL_PDFS               = 20     // max local redact-test PDFs to include
```

## PDF Corpus

The corpus is managed by `PdfCorpus.java`:

- **Downloaded corpus:** 30+ PDFs from Mozilla pdf.js test suite (annotations, bugs,
  forms, CJK text, images, etc.)
- **Local corpus:** 84 generated test PDFs covering fonts, transforms,
  spacing, ligatures, rendering modes, etc.
- **Cache:** Downloaded PDFs are cached in a temp directory. If download fails, the
  test continues with cached versions.

### Regenerating Test PDFs

```bash
./gradlew :jpdfium:run -PmainClass=stirling.software.jpdfium.redact.RedactTestPdfGenerator
```

This regenerates all 84 `redact-test-*.pdf` files in `src/test/resources/pdfs/redact/`.

## Interpreting the HTML Report

The report at `samples-output/corpus-redact-report/index.html` is organized as:

1. **Summary banner** -- total counts by severity
2. **Threshold reference** -- collapsible table showing current detection thresholds
3. **PDF cards** -- one per PDF, sorted: FAIL first, then WARN, then PASS
   - Each card shows: page count, match count, issues list
   - Metrics table: chars before/after, char loss %, pixels changed %, missing words
   - Visual diffs: before/after/diff images (expanded for FAIL/WARN, collapsed for PASS)

### What to look for

- **FAIL cards:** These need investigation. Check if the Object Fission algorithm
  corrupted the PDF structure, or if the PDF has unusual encoding.
- **WARN cards:** Manual inspection recommended. Open the before/after images and
  verify that only the target words were visually removed.
- **PASS cards:** No action needed. The diff images are still available if you want
  to spot-check.

## Adding New PDFs to the Corpus

### Downloaded corpus
Add URLs to the `URLS` list in `PdfCorpus.java`. Use raw GitHub URLs from the
`mozilla/pdf.js` test repository.

### Local test PDFs
Add generation methods to `RedactTestPdfGenerator.java`. The `save()` method
automatically places files in `src/test/resources/pdfs/redact/`.

## Troubleshooting

### OOM during tests
The integration test runs with 2GB heap (`maxHeapSize = "2g"` in `build.gradle.kts`).
If you add many more PDFs or increase DPI, you may need more memory.

### Tests fail to find resources
PDFs are expected at `/pdfs/general/` and `/pdfs/redact/` on the test classpath.
Check that `src/test/resources/pdfs/` exists and contains the expected files.

### Download failures
If `PdfCorpus.download()` fails (e.g., no internet), the test falls back to cached
PDFs. First-run requires internet. Subsequent runs use the cache.
